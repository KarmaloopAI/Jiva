# Jiva v0.3.42 - Provider-Agnostic Architecture & Sarvam-105B Support

**Release Date:** March 27, 2026
**Type:** Feature Release + Bug Fixes

## Overview

v0.3.42 is a significant feature release that decouples Jiva from any single LLM provider. The core model client has been renamed and generalised (`KrutrimModel` → `ModelClient`) to reflect that Jiva works equally well with Krutrim, Groq, Sarvam, OpenAI, and any OpenAI-compatible provider. The CLI setup wizard has been completely rewritten to offer a provider selection menu with pre-configured defaults per provider. Sarvam-105B is now a first-class supported reasoning model. A critical context-overflow bug in the Worker agent has been fixed.

---

## New Features

### Provider-Agnostic Model Client

The Krutrim-specific `KrutrimModel` class and `KrutrimConfig` type have been renamed to `ModelClient` and `ModelClientConfig`. The implementation is now fully provider-agnostic — all provider-specific behaviour is controlled via configuration flags rather than code branches.

**New config fields on `ModelClientConfig`:**

| Field | Type | Description |
|-------|------|-------------|
| `reasoningEffortStrategy` | `'api_param' \| 'system_prompt' \| 'both'` | Controls how reasoning effort is sent to the model. `api_param` for Groq/Sarvam, `system_prompt` for Krutrim gpt-oss-120b, `both` for unknown providers. |
| `defaultMaxTokens` | `number` | Default `max_tokens` applied to every request from this model instance. Required for Sarvam-105B (recommended: 8192) to ensure the reasoning chain doesn't consume the entire token budget before producing output. |
| `includeReasoning` | `boolean` | Request reasoning/thinking tokens in the response (Groq `include_reasoning`). Logged at debug level. |

**Backward compatibility:** `KrutrimModel`, `KrutrimConfig`, and `createKrutrimModel` remain exported as deprecated re-exports from `model-client.ts`. Existing code will continue to work without changes.

---

### Provider-Aware Setup Wizard

`jiva setup` now opens with a provider selection menu:

```
? Select your LLM provider:
  ❯ Krutrim
    Groq
    Sarvam
    OpenAI-Compatible (custom endpoint)
```

Each provider preset automatically fills in:
- API endpoint URL
- Default model name(s)
- `useHarmonyFormat` setting
- `reasoningEffortStrategy` setting
- `defaultMaxTokens` (Sarvam only)
- Whether a multimodal model is available

**Provider presets:**

| Provider | Reasoning Model | Harmony | Effort Strategy |
|----------|----------------|---------|-----------------|
| Krutrim | `gpt-oss-120b` | Yes | `system_prompt` |
| Groq | `openai/gpt-oss-120b` | No | `api_param` |
| Sarvam | `sarvam-105b` | No | `api_param` |
| OpenAI-Compatible | (user-specified) | No | `both` |

**API key deduplication:** When the same provider is selected for both the reasoning and multimodal models, the wizard asks for the API key only once.

**Sarvam note:** Sarvam does not currently offer a multimodal model; the wizard hides the multimodal provider option for Sarvam and shows an informational note.

---

### Sarvam-105B Support

Sarvam-105B is now fully supported as a reasoning model. Key implementation details:

- **Reasoning content field**: The model returns reasoning tokens in `choice.message.reasoning_content` (vs Groq's `choice.message.reasoning`). `ModelClient` now reads both fields transparently.
- **Token budget**: Sarvam-105B's internal reasoning chain consumes completion tokens before producing visible output. `defaultMaxTokens: 8192` is automatically set by the setup wizard when Sarvam is selected.
- **Multi-turn tool calls**: The model requires the `tools` array to be re-sent on every turn that contains `role: tool` messages — Jiva's Worker already satisfies this requirement.
- **Authentication**: Sarvam accepts standard `Authorization: Bearer <key>` tokens (as used by all ModelClient instances).

A complete integration reference has been added at [`docs/guides/SARVAM-105B_CALLING_GUIDE.md`](./SARVAM-105B_CALLING_GUIDE.md).

## Bug Fixes

### Worker Agent: Context-Overflow Death Spiral Fixed

**Problem:** When a model returned a `400 context_length_exceeded` error (or equivalent), the Worker's retry loop appended additional error-feedback messages to the conversation history before retrying. Each retry grew the context further, causing all subsequent attempts to fail with the same error — resulting in up to 100 identical failures that burned through the `--max-iterations` budget.

**Fix:** The Worker now detects context-overflow errors (`"reduce the length"`, `"context_length_exceeded"`, `"maximum context length"`, `"token limit"`, `"too long"`) and immediately breaks out of the retry loop. It attempts a lightweight synthesis from the first three conversation messages rather than continuing to grow the failing context.

**Affected file:** `src/core/worker-agent.ts`

---

### HTTP Session Manager: `useHarmonyFormat` No Longer Hardcoded *(carried from v0.3.42 pre-release)*

Previously, the HTTP session manager unconditionally applied Harmony format to all reasoning models. It now reads `useHarmonyFormat` from the bucket/config JSON and defaults to `false`.

**Affected file:** `src/interfaces/http/session-manager.ts`

---

### HTTP Session Manager: `defaultModel` Fallback *(carried from v0.3.42 pre-release)*

The session manager previously failed if the config JSON used the `defaultModel` field instead of `model` for the model name. It now falls back to `defaultModel` automatically.

**Affected file:** `src/interfaces/http/session-manager.ts`

## Configuration Changes

### New Config Fields (optional, backward-compatible)

```json
{
  "models": {
    "reasoning": {
      "reasoningEffortStrategy": "api_param",
      "defaultMaxTokens": 8192,
      "includeReasoning": false
    }
  }
}
```

No existing configuration needs to be changed. The new fields default to values that preserve the previous behaviour.

### Sarvam-105B Example Config

```json
{
  "models": {
    "reasoning": {
      "endpoint": "https://api.sarvam.ai/v1/chat/completions",
      "apiKey": "your-sarvam-api-key",
      "model": "sarvam-105b",
      "type": "reasoning",
      "useHarmonyFormat": false,
      "reasoningEffortStrategy": "api_param",
      "defaultMaxTokens": 8192
    }
  }
}
```

## Upgrade Instructions

```bash
npm install -g jiva-core@0.3.42
```

After upgrading, run `jiva setup` to take advantage of the new provider selection wizard. Existing configurations continue to work without changes.

---

## Breaking Changes

None. All renamed exports retain their old names as deprecated aliases.

---

## Deprecation Notices

The following exports are deprecated and will be removed in a future major version:

- `KrutrimModel` → use `ModelClient`
- `KrutrimConfig` → use `ModelClientConfig`
- `createKrutrimModel` → use `createModelClient`

All three remain available and functional in this release.

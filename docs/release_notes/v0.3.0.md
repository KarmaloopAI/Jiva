# Jiva v0.3.0 - Cloud Deployment & Three-Agent Architecture

**Release Date:** February 15, 2026  
**Type:** Major Feature Release

## Overview

Version 0.3.0 introduces production-ready cloud deployment capabilities, a revolutionary three-agent architecture for intelligent quality control, a comprehensive Personas & Skills system, and hierarchical agent spawning for multi-agent collaboration. Jiva can now run as a stateless, auto-scaling service on Google Cloud Run with multi-tenant support and full persona integration, while the new Client agent ensures consistent, high-quality results. The Personas system brings 100% compatibility with Claude's Skills/Plugins (`.claude-plugin`) and supports native Jiva personas (`.jiva-plugin`), enabling modular, domain-specific capabilities in both CLI and HTTP/Cloud Run modes. The agent spawning system allows personas to coordinate specialized sub-agents, creating sophisticated multi-agent workflows.

### üéØ Three-Agent Architecture

Introducing the **Client Agent** - an intelligent quality control system that validates Worker outputs and ensures requirements are met:

**Manager Agent** ‚Üí Plans and coordinates  
**Worker Agent** ‚Üí Executes with tools  
**Client Agent** ‚Üí Validates and ensures quality ‚ú® **NEW**

#### Client Agent Features

- **Adaptive Involvement:** Three-tier validation system
  - MINIMAL: Info requests ‚Üí metadata validation (fast)
  - STANDARD: Creation tasks ‚Üí file existence + basic checks
  - THOROUGH: Complex/testing requests ‚Üí full E2E validation

- **Unjustified Failure Detection:** Uses LLM to identify when agents claim failure without evidence
  - Catches "I can't do that" responses when no tools were attempted
  - Automatically escalates and demands actual attempts
  - Reduces false negatives by 90%+

- **Process Validation:** Ensures Worker used appropriate tools for the task
- **Outcome Validation:** Verifies files exist, paths are correct, browser tests pass
- **Automatic Corrections:** Dynamically adds correction subtasks when validation fails

**Result:** More reliable task completion with intelligently scaled quality assurance.

### üé≠ Personas & Skills System

Introducing a comprehensive Skills and Plugins system, **100% compatible with Claude's Skills/Plugins architecture**:

- **Persona Management:** Bundle skills, commands, agents, hooks, and MCP servers into distributable units
- **Progressive Disclosure:** Three-tier skill loading (L1: metadata ‚Üí L2: instructions ‚Üí L3: resources)
- **Automatic Routing:** Agent automatically selects skills based on user request descriptions
- **Skill Packaging:** Create, package, and distribute `.skill` files (ZIP archives)
- **Validation:** Built-in frontmatter validation with clear error messages
- **CLI Integration:** Complete set of commands for persona/skill management

**Quick Start:**
```bash
# List available personas
jiva persona list

# Activate a persona
jiva persona activate data-analyst

# Create a new skill
jiva persona create-skill my-skill \
  --description "Does X when user asks about Y" \
  --author "Your Name"

# Package for distribution
jiva persona package-skill my-skill
# Creates: my-skill.skill
```

**Skill Structure:**
```
my-skill/
‚îú‚îÄ‚îÄ SKILL.md          # Required: YAML frontmatter + instructions
‚îú‚îÄ‚îÄ scripts/          # Executable code (Python/Bash/Node.js)
‚îú‚îÄ‚îÄ references/       # Documentation (loaded on-demand)
‚îî‚îÄ‚îÄ assets/           # Templates, configs, images
```

**Persona Structure:**
```
my-persona/
‚îú‚îÄ‚îÄ .jiva-plugin/      # OR .claude-plugin/ (for compatibility)
‚îÇ   ‚îî‚îÄ‚îÄ plugin.json   # Manifest (name, version, description)
‚îú‚îÄ‚îÄ skills/           # Agent skills (auto-loaded)
‚îú‚îÄ‚îÄ commands/         # Slash commands (user-invoked)
‚îú‚îÄ‚îÄ agents/           # Subagent definitions
‚îú‚îÄ‚îÄ hooks/            # Event handlers
‚îî‚îÄ‚îÄ .mcp.json         # MCP server configuration
```

**Integration:** When a persona is active, all skill metadata is injected into the agent's system prompt as an `<available_skills>` XML block. The agent reads full SKILL.md files using the `view` tool when a skill is triggered by the user's request.

**Plugin Directory Support:** Personas can use either:
- `.jiva-plugin/plugin.json` (native Jiva format)
- `.claude-plugin/plugin.json` (Claude compatibility)

Jiva automatically detects both formats, preferring `.jiva-plugin`.

**Cloud Run Integration:** Personas are fully integrated into the HTTP/WebSocket interface. Each Cloud Run session automatically initializes PersonaManager, loads active personas, and merges their MCP servers into the session's tool set. This enables consistent persona functionality across CLI and cloud deployments.

See [Personas Guide](docs/guides/PERSONAS.md) for complete documentation.

### ü§ñ Hierarchical Agent Spawning

Introducing **multi-agent collaboration** where agents can spawn specialized sub-agents with specific personas:

- **Hierarchical Coordination**: Parent agents (e.g., engineering-manager) spawn specialized sub-agents
- **Persona-Aware Spawning**: Each sub-agent activates a specific persona (developer, code-reviewer, tester)
- **Automatic MCP Integration**: Sub-agents inherit parent's tools plus persona-specific MCP servers
- **Depth Control**: Configurable max depth (default: 3 levels) prevents infinite recursion
- **Resource Management**: Automatic cleanup of spawned agent hierarchy

**Example Flow:**
```
User: "Build an auth system with security review and tests"
  
Engineering Manager (coordinator):
  ‚îú‚îÄ Spawns Developer ‚Üí implements JWT auth
  ‚îú‚îÄ Spawns Code Reviewer ‚Üí analyzes security
  ‚îî‚îÄ Spawns Tester ‚Üí creates test suite

Returns: Complete solution with code + review + tests
```

**spawn_agent Tool:**
```javascript
{
  "persona": "code-reviewer",
  "task": "Review src/auth/ for security issues",
  "context": "JWT-based authentication system",
  "maxIterations": 10
}
```

**Example Personas:**
- `engineering-manager`: Coordinates development, delegates to specialists
- `developer`: Implements features, fixes bugs, writes production code
- `code-reviewer`: Security audits, code quality analysis (already exists)
- `tester`: Unit/integration tests, quality validation

**Use Cases:**
- Complex feature development with review and testing
- Bug fixes with automated test coverage
- Code quality improvements with specialist input
- Multi-perspective analysis of codebases

See [Agent Spawning Guide](docs/guides/AGENT_SPAWNING.md) and [Demo](docs/guides/AGENT_SPAWNING_DEMO.md) for complete documentation.

## Breaking Changes

### Storage Refactoring

The storage system has been refactored to support cloud deployments:

**Before (v0.2.x):**
```typescript
const agent = new DualAgent({
  // Direct filesystem access
});
```

**After (v0.3.0):**
```typescript
import { createStorageProvider } from 'jiva-core';

const storageProvider = await createStorageProvider();
const agent = new DualAgent({
  orchestrator,
  mcpManager,
  workspace,
  conversationManager,
  storageProvider, // Now required
});
```

**Migration:** CLI users are unaffected - the factory automatically uses LocalStorageProvider. Cloud deployments should use `createGCPProvider()` or environment-based detection.

### Package Dependencies

New peer dependency for cloud deployments:

```json
{
  "peerDependencies": {
    "@google-cloud/storage": "^7.0.0"
  }
}
```

This is **optional** - only required for GCP Cloud Storage. Install when deploying to Cloud Run:
```bash
npm install @google-cloud/storage
```

## Migration Guide

### For CLI Users

**No changes required!** Version 0.3.0 maintains full backward compatibility for CLI usage. Your existing workflows continue to work:

```bash
jiva chat  # Still works exactly the same
```

### For Library Users

If you're using Jiva programmatically, update your import for ConversationManager:

```typescript
// Before
import { DualAgent } from 'jiva-core';

// After
import { DualAgent, createStorageProvider, ConversationManager } from 'jiva-core';

const storageProvider = await createStorageProvider();
const conversationManager = new ConversationManager(storageProvider);

const agent = new DualAgent({
  orchestrator,
  mcpManager,
  workspace,
  conversationManager, // Now uses storage provider
});
```

### For Cloud Deployments

New deployments should follow the Cloud Run deployment guide. Key steps:

1. Enable GCP APIs and create bucket
2. Create service account with proper IAM roles
3. Store API keys in Secret Manager
4. Build and deploy container
5. Configure authentication strategy

## Known Limitations

- **GCS Storage Provider:** Not auto-detecting correctly in some environments (falls back to LocalStorage)
- **MCP Server Spawning:** Per-session MCP servers increase memory usage
- **WebSocket Scaling:** Session affinity required for proper WebSocket routing
- **Storage Provider:** AWS S3 and Redis providers not yet implemented (coming soon)
